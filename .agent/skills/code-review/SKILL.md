# コードレビュースキル

## 概要

未コミットの変更、または指定されたファイルに対してコードレビューを実行します。

## レビュー観点

### 1. TypeScript型安全性（Frontend）
- `any` の使用がないか
- 不要な型アサーション（`as`）がないか
- デフォルトエクスポートを使っていないか（named export のみ）
- `lib/types.ts` や `types/judging.ts` の型定義が適切に使われているか

### 2. Next.js App Router規約（Frontend）
- Server / Client Componentsの使い分けが適切か
- `'use client'` が必要最小限のコンポーネントにのみ付いているか
- ページは `page.tsx`、レイアウトは `layout.tsx` に配置されているか
- メタデータが適切に設定されているか

### 3. スタイル（Frontend）
- Tailwind CSS 3.4のユーティリティが適切に使われているか
- マジックナンバーがないか
- レスポンシブはモバイルファーストで書かれているか

### 4. API通信（Frontend）
- `lib/api.ts` の API 関数オブジェクト（`contestApi`, `entryApi` 等）を使っているか
- 直接 `axios` を呼んでいないか
- エラーハンドリングが適切か
- ローディング状態が考慮されているか
- React Query（TanStack）のパターンに従っているか

### 5. Django / DRF 規約（Backend）
- モデルに `verbose_name` と docstring が付いているか
- ViewSet のパターン（`get_serializer_class`, `get_permissions`）に従っているか
- Django ORM を使い、SQL 文字列結合をしていないか
- `except Exception` で広範囲に例外を握りつぶしていないか
- 型ヒントが関数シグネチャに付いているか

### 6. セキュリティ
- APIキー・シークレットがハードコードされていないか
- ユーザー入力のバリデーションが適切か
- 権限チェックが漏れていないか

### 7. テスト
- 新機能に対応するテストがあるか
- テスト名が「条件 + 期待する結果」で書かれているか
- 既存テストが削除・弱体化されていないか

## 実行手順

1. `git diff` で変更差分を取得する
2. 変更されたファイルを一つずつ確認する
3. 上記の観点でレビューコメントを作成する
4. 問題の深刻度を分類する:
   - 🔴 **Critical**: 必ず修正が必要（型安全性の破壊、セキュリティ問題、権限チェック漏れ）
   - 🟡 **Warning**: 修正を推奨（パターンの不一致、テスト不足）
   - 🔵 **Info**: 改善提案（可読性向上、リファクタリング候補）
5. レビュー結果をまとめて報告する

## 出力フォーマット

```markdown
## コードレビュー結果

### 変更サマリ
- 変更ファイル数: X
- 追加行数: +Y
- 削除行数: -Z

### 指摘事項

#### 🔴 Critical
- [ファイル名:行番号] 指摘内容

#### 🟡 Warning
- [ファイル名:行番号] 指摘内容

#### 🔵 Info
- [ファイル名:行番号] 指摘内容

### 総評
全体的な品質評価と改善提案
```
